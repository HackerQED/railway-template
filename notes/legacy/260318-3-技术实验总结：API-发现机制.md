# 技术实验总结：API 发现机制

**日期**：2026-03-18
**结论**：核心假设验证通过，可进入应用开发
**实验仓库**：https://github.com/HackerQED/cc-music-video-playground

---

## 实验了什么

验证一个核心问题：**Agent 只拿到一个 URL，能不能自主发现我们的 API 并正确调用？**

具体来说，搭了一个 Next.js mock 服务（6 个 API 端点），配上三层发现机制（llms.txt、Skill、文档端点），然后让 Claude Code 在零准备的情况下从 URL 出发，完成 Music Video 生成的完整编排流程。

### 验证场景（三级递进）

| Level | 场景 | 结果 |
|-------|------|------|
| **1** | 预装 Skill，给明确任务 | ✅ 通过 |
| **2** | **只给 URL，无任何准备**（核心验收） | ✅ 通过 |
| **3** | 发现 llms.txt 里故意隐藏的能力 | 场景搭建完成 |

Level 2 是核心验收标准——真实用户不一定会装 Skill，但只要给 Agent 一个 URL，它就应该能自己搞定。

### Level 2 实测记录（Claude Code，不装 Skill，干净工作空间）

1. **「http://localhost:3000 能做视频，怎么做？」** → Agent 自主访问站点，读 HTML 发现能力，总结出 4 步流程
2. **「用它做一个音乐视频。音乐地址是 https://example.com/song.mp3」** → 完成 analyze → image → video → compose 全流程
3. **「用这个生成一首歌」** → Agent 只看了首页 HTML 就判断「不能」，没有主动去查 llms.txt 和 /api/capabilities。被追问后才去查

第 3 条暴露了一个重要问题：Agent 不一定会读 llms.txt，它可能只看 HTML 就下结论。

## 选择了什么路线

### 产品定位：Human First, Agent Ready

不做 Agent 产品，做一个普通人能用的表单生成站，在此基础上加 AI 友好的 API + Skill。

```
REST API（核心，人和机器都调这个）
   ├── 表单前端（给人用）
   ├── Skill 描述（给 Agent 发现用，轻量级）
   └── MCP Server（可选，不优先）
```

**为什么不做 Chat 界面？** 竞品调研发现：用 Chat 界面的竞品（Suno 等）用户抱怨远多于用表单界面的 B.ai。Chat 界面引入的问题（打圈圈、不可干预、不稳定）远大于它带来的价值。先做表单，赚到钱再加 Chat 作为营销手段。

**为什么不做 MCP？** MCP 在 2026 年初已有大量"MCP is dead"讨论。Code-first 方式比 MCP 省高达 98% token，且 OpenClaw、Pi 等平台不支持 MCP。Skills 是现在更成熟的方案。

### API 发现机制：三层递进

不一次性灌全部文档，而是按需发现、逐层深入。

**第一层：llms.txt**（零成本入口）
- 放在网站根目录 `GET /llms.txt`，纯文本 Markdown
- Agent 访问网站时自动读取
- 只写摘要，指向 `/api/capabilities` 获取完整列表
- 推荐安装 Skill

**第二层：Skill**（加速器，非必须）
- 一个 SKILL.md 文件，兼容 Claude Code 和 OpenClaw
- 包含 API 基础 URL、工作流说明、端到端示例
- 通过 `GET /skill/SKILL.md` 下载

**第三层：API 文档端点**（按需加载）
- `GET /api/capabilities` — 能力列表（权威数据源）
- `GET /api/docs/:capability` — 单个能力的 Markdown 文档

**没选 OpenAPI Spec**：Agent 天然擅长读 Markdown，不是 YAML/JSON。能解析 OpenAPI 的主要是框架层（Dify、Semantic Kernel），不是 Agent 自己。上生产接 Dify 时再加不迟。

### 错误处理设计（对 Agent 至关重要）

两个特殊设计：

1. **每个错误响应带 `doc` 链接**：让 Agent 永远有地方可以去寻址
```json
{
  "error": "missing required field 'audio_url' (string, URL of the audio file)",
  "doc": "/api/docs/music-analyze"
}
```

2. **全局 404 兜底**：Agent 走到不存在的路径时，列出所有可用端点，引导回到正轨
```json
{
  "error": "endpoint not found",
  "doc": "/api/capabilities",
  "available_endpoints": ["GET /api/capabilities", "GET /api/docs/:capability", "..."]
}
```

### 跨 Agent 兼容

设计上不依赖任何单一 Agent 供应商。Skill 格式同时兼容 Claude Code 和 OpenClaw。OpenClaw 的 ClawHub（13,729 个社区 skill）是重要分发渠道。

## 注意事项

### 1. 所有信息源必须指向同一个 ground truth

主页 HTML、llms.txt、/api/capabilities、Skill——这些都是同一套接口信息的不同呈现。如果各自硬编码，改一个忘改另一个，Agent 就会拿到矛盾的信息做出错误判断。

**做法**：
- `lib/api-helpers.ts` 的 `CAPABILITIES` 数组是权威来源
- `lib/docs.ts` 是所有文档的唯一来源
- llms.txt 和主页只写摘要，不列完整接口列表，指向 `/api/capabilities`
- 人看的文档和机器看的文档数据源不统一，维护成本会指数级增长

### 2. HTML 和 llms.txt 信息必须对齐

实测发现 Agent 有时只看 HTML 不看 llms.txt（反之亦然）。关键信息两边都要有：
- 能力概览
- Skill 安装入口
- 指向 /api/capabilities 的引用

### 3. 批量化 API 对 Agent 体验很重要

实测观察到 Agent 串行调用 6 次 `video/generate`，真实 API 下体验会很差。需要两层批量化：
1. **批量提交**：一次传多个任务，减少工具调用次数
2. **批量轮询**：统一的 `/api/tasks/status` 接口，传一组 task_id 一次返回所有状态

方向明确，不需要单独验证，直接做。

### 4. Agent 不会主动深挖

Level 2 第 3 条测试暴露：Agent 看了首页 HTML 列出的 4 个能力就判断「不能生成音乐」，没有主动去查 llms.txt 或 /api/capabilities 验证。

**启示**：不能假设 Agent 会穷尽所有信息源。关键能力必须在最显眼的位置（HTML + llms.txt）都露出，不能藏在需要额外请求才能发现的地方。

### 5. Skill 里写生产 URL，本地开发用环境变量覆盖

Skill 正文直接写生产地址（如 `https://your-domain.com`），提示可通过 `MEDIA_API_BASE_URL` 环境变量覆盖。不过度设计。

## 实验中的 API 设计参考

| 端点 | 方法 | 说明 |
|------|------|------|
| `/llms.txt` | GET | Agent 入口，站点概览（纯文本 Markdown） |
| `/skill/SKILL.md` | GET | Skill 文件下载 |
| `/api/capabilities` | GET | 所有能力列表（权威数据源） |
| `/api/docs/:capability` | GET | 单个能力的 Markdown 文档 |
| `/api/music/analyze` | POST | 音乐分析（BPM、情绪、分段） |
| `/api/music/generate` | POST | 音乐生成 |
| `/api/image/generate` | POST | 文字生图 |
| `/api/video/generate` | POST | 图转视频 |
| `/api/video/compose` | POST | 拼合最终视频 |
| `/api/*`（其他） | ANY | 404 兜底，返回所有可用端点 |

## 后续开发建议

1. **API 层**：把 mock 换成真实供应商（复用之前的生视频、生图代码）
2. **批量化**：加批量提交和批量轮询接口
3. **前端**：做表单界面（Human First）
4. **鉴权**：MVP 阶段先用 API key 硬编码，后续再做完整 auth 流程
5. **分发**：把 Skill 发布到 OpenClaw ClawHub
6. **OpenAPI**：接 Dify 等平台时再加
