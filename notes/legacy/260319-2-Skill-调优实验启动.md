## 背景

实验1（API 发现机制）和实验2（Skill 设计研究）已在 playground 仓库完成。现在进入主仓库，用真实 API 写真正的 Skill 并调优。

**Yino AI** 是 All-in-One 的媒体生成平台。MV 是 marketing trigger 和主页关键词，但底层是通用的生图/生视频能力。Skill 是给 Agent 的皮，Web App 是给人的皮，API 是核心。

**本轮实验目标：写出能用的 Skill，不是做对比实验。** 这是可行性验证——把代码写出来，调优到 Agent 能正确完成任务。

## 核心研究问题

### 1. 跨平台兼容

不同 Agent 平台的 Skill 安装路径不同：
- Codex: `~/.codex/skills/<name>/`
- Claude Code: `.claude/skills/<name>/`（项目级）或 `~/.claude/skills/<name>/`（全局）
- OpenClaw: `~/.openclaw/workspace/skills/<name>/`

硬编码路径就会不兼容。参考样本 nano-banana-pro 的 Codex 版本就是这个问题——路径写死了 `~/.codex/`。

**初步做法**：用相对描述（"in the `scripts/` directory next to this SKILL.md file"），让 Agent 自己定位。比硬编码更鲁棒。

### 2. Skill 是否需要带代码

两个参考样本都带了代码：
- **nano-banana-pro** — 带 Python 脚本（`generate_image.py`），用 `uv run` 执行
- **free-ride** — 带 CLI 工具（`pip install -e .`）

问题：我们的 yino-music-video Skill 是否需要带代码？还是纯靠 `curl` / HTTP 调用我们的 REST API 就够了？

带代码的好处：可以封装复杂逻辑（如轮询异步任务状态、拼接多步调用）。
纯 API 的好处：跨平台兼容性最好，不依赖运行时环境。

### 3. 渐进式披露

和实验1的 llms.txt 设计一脉相承——Skill 里放多少信息？

参考样本的做法：
- **Preflight checklist** — 先检查环境，快速失败
- **Primary workflow 放最前面** — Agent 最常走的路径要最容易找到
- **按需深入** — Skill 写摘要，引导 Agent 去 `/api/docs/:capability` 查完整参数

关键：Skill 是给 Agent 读的，不是人的文档。精简 > 全面。

### 4. 长工作流编排

MV 生成是多步链（analyze → image × N → video × N → compose），Skill 怎么描述才能让 Agent 正确编排？需要验证：
- Agent 能否理解步骤间的数据依赖
- 批量操作（N 张图、N 段视频）怎么描述
- 失败重试和部分完成怎么处理

## 已验证的结论

### nano-banana-pro 在 Claude Code 上能跑通

把 nano-banana-pro 装到 `.claude/skills/nano-banana-pro/`，改掉硬编码路径后，Claude Code 能正常触发并执行。

**验证了的事情：**
- OpenClaw 生态的 skill（带 Python 脚本的）可以移植到 Claude Code
- 相对路径方案可行：SKILL.md 里不写任何绝对路径，只描述"脚本在 `scripts/` 目录下"，让 Agent 自己找
- `uv run` 作为跨平台 Python 执行方式是通用的（不需要 pip install）

**各平台 skill 路径变量对比：**
- Claude Code: `${CLAUDE_SKILL_DIR}` — 内置变量，替换为 SKILL.md 所在目录
- OpenClaw: `{baseDir}` — 同样功能
- 但两者语法不通用，所以用自然语言描述 + 让 Agent 自己找是最安全的

### free-ride 不适用于 Claude Code

free-ride 是给 OpenClaw 换免费 LLM 模型的工具，完全依赖 OpenClaw 基础设施（`~/.openclaw/openclaw.json`、`openclaw gateway restart`）。跟 Claude Code 无关。

**但它暴露了一个分类：** OpenClaw skill 分两种——
1. **通用 skill**（调外部 API 做事）— 天然跨平台，如 nano-banana-pro、api-gateway
2. **平台 skill**（配置 OpenClaw 自身）— 不可移植，如 free-ride

我们的 yino-music-video 属于第 1 类。

### Skill 生态现状（截至 2026.03）

- ClawHub 有 13,729 个社区 skill，是目前最大的公开 Skill 注册表
- Claude Code 没有集中的 skill 市场，skill 通过项目目录或 `~/.claude/skills/` 手动安装
- OpenCode 兼容 `.claude/skills/` 路径，也兼容 `.opencode/skills/` 和 `.agents/skills/`
- Claude Code 的 skill 遵循 [Agent Skills](https://agentskills.io) 开放标准
- Snyk 审计发现 ClawHub 13.4% 的 skill 有安全问题，341 个在偷数据 — 发布时要注意信任度

### 核心设计决策：不带代码，接口层和知识层分离

**结论：yino 的 skill 不带代码。** 纯 instruction-only，让 agent 自己用 curl/fetch/requests 调 API。

**理由：**
- 跨平台兼容性最好——能跑在 Claude Code、OpenClaw、GPT、网页沙箱、任何有 HTTP 能力的地方
- Agent 自己会写代码。如果某个任务太重复，agent 可以自己写脚本来解决，用什么语言它自己选
- 异步轮询等问题可以在 API 侧解决（长轮询、webhook），不需要 skill 侧写代码
- 实验1已验证：agent 拿到 API 说明后能自主完成整个 MV 编排流程

**Skill 内部分两层：**

```
yino-music-video/
├── SKILL.md                    # 接口层：API 怎么调（端点、参数、流程）
└── knowledge/
    ├── music-analyze.md        # 知识层：怎么解读音乐分析结果
    ├── image-prompt-craft.md   # 知识层：怎么根据音乐写好视觉 prompt
    ├── video-motion.md         # 知识层：运镜和动态选择
    └── compose-timing.md       # 知识层：剪辑节奏和转场
```

**接口层（SKILL.md）** — 怎么调 API。API 变了改这里。
**知识层（knowledge/）** — 怎么做得好。想提升 MV 质量改这里，不碰任何代码。

**关键洞察：调优 skill 的质量 = 调优知识文档的质量，不是调代码。**

这意味着：
- 我们可以随时更新领域知识来提升工作流质量，而不需要发版
- 用户可以注入自己的领域知识（往 knowledge/ 里加文件），定制自己的审美偏好
- 知识和接口各自独立演进，互不干扰

**产品线拆分与实验顺序：**

| Skill | 定位 | 范围 |
|-------|------|------|
| **yino-ai** | 通用入口，调用 Yino 平台所有能力 | 轻量，只有 SKILL.md，到处能插 |
| **yino-music-video** | MV 生成完整编排 | SKILL.md（接口层 + 通用编排知识）+ knowledge/（深度领域知识） |

**实验顺序：先 yino-ai，再 yino-music-video。**

1. 先写 yino-ai，验证纯 API 调用能搞定
2. 在它的基础上写 yino-music-video，叠加领域知识

yino-music-video 本质上就是 yino-ai 的加强版——同一个接口层，加了编排知识和领域知识。

**yino-music-video 的 SKILL.md 三段式结构：**

```
# 第一部分：接口层（跟 yino-ai 一致）
怎么调 API，端点，参数...

# 第二部分：MV 编排通用知识（直接写在 SKILL.md 里，不另开文件）
- 整体流程：analyze → image × N → video × N → compose
- 每一步的核心原则
- 步骤间数据怎么传递

# 第三部分：知识指引（指向 knowledge/ 目录）
深入某一步时，参考对应的知识文档：
- [knowledge/music-analyze.md] — 怎么解读音乐分析结果
- [knowledge/image-prompt-craft.md] — 怎么写好视觉 prompt
- ...
```

入口知识直接放 SKILL.md 里而不是另开文件，因为实验1验证过：agent 不会主动深挖，关键信息必须在最显眼的位置。knowledge/ 里的文件是"想了解更多"的深度内容，agent 按需读取。

## 主仓库现状（2026-03-19）

### 已有的真实 Agent API

| 端点 | 类型 | 说明 |
|------|------|------|
| `POST /api/agent/models/seedream-4-5` | 异步 | 文本 → 图片（KIE 后端），支持批量 |
| `POST /api/agent/models/veo-3` | 异步 | 图片 → 视频（KIE Veo 后端），支持批量 |
| `POST /api/agent/music/analyze` | 同步 | 音乐分析（OpenRouter + Gemini 3.1 Pro） |
| `POST /api/agent/video/compose` | 同步 | FFmpeg 视频合成 + S3 上传 |
| `GET /api/agent/generations/status` | 查询 | 批量轮询异步任务状态 |
| `POST /api/agent/upload` | 同步 | 文件上传（图片 4MB / 音频 50MB） |
| `GET /api/agent/capabilities` | 公开 | 能力发现（JSON） |
| `GET /llms.txt` | 公开 | Agent 入口（纯文本） |

鉴权：API Key（`Authorization: Bearer yino_xxx`）+ Session Cookie 双轨。

## 执行计划

### Phase 1：yino-ai Skill（通用入口）

**目标**：写一个轻量的通用 Skill，验证"纯 instruction + API 调用"模式能跑通。

**范围**：只有一个 SKILL.md，覆盖所有当前能力（生图、生视频、音乐分析、视频合成），不含编排知识。

**验证方式**：
1. 装到 Claude Code（`.claude/skills/yino-ai/`）
2. 让 Agent 完成单项任务：生一张图、分析一首歌
3. 确认鉴权、批量提交、异步轮询都能正确工作

**交付物**：`skills/yino-ai/SKILL.md`

**目标**：在 yino-ai 基础上叠加 MV 编排知识和领域知识，让 Agent 能完成完整的 MV 生成工作流。

**结构（三段式）**：

```
skills/yino-music-video/
├── SKILL.md                    # 接口层 + MV 编排通用知识

    ├── image-prompt-craft.md   # 怎么根据音乐写视觉 prompt
    ├── video-motion.md         # 运镜和动态选择
    └── compose-timing.md       # 剪辑节奏和转场
```

**验证方式**：
1. 装到 Claude Code
2. 给一首真实歌曲，让 Agent 完成完整 MV 流程
3. 调优知识文档直到输出质量可接受

## Playground 实验结论摘要

以下结论来自 `cc-music-video-playground` 仓库的实验1和实验2，不再重复展开：

**设计决策：**
- [x] 不带代码，纯 instruction-only — 跨平台兼容性最好
- [x] 不硬编码路径 — 用自然语言描述让 Agent 自己找
- [x] 接口层和知识层分离 — 调优质量 = 调优知识文档
- [x] 产品线拆分 — yino-ai（通用）→ yino-music-video（加强版）
- [x] 关键信息必须在 SKILL.md 最显眼位置 — Agent 不会主动深挖

**API 发现机制：**
- [x] llms.txt + /api/capabilities + /api/docs 三层渐进披露
- [x] 所有数据源从同一个 CAPABILITIES 注册表生成（已在主仓库落地）
- [x] 批量化提交 + 批量化轮询（已在主仓库落地）
