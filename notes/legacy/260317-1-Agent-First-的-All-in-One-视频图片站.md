先说答案，我接下来可能想做一个 Agent First 的 All in One 视频图片站。

就是一个普通的视频图片站，我所有学过的东西都可以复用，只是说加一个 skill，可以调用我的 API。随便用户用什么 OpenClaw，用 Cloud Code 调。

这个想法，以及下面的一些想法，都是来自于我最近对 Music Video 这个品类的三个竞队的调研。这些想法背后都是有内在逻辑的，我尝试在这里说明一下：

---

我在做了 30 天的 Agent 之后思考到的事情就是：
正确的做 Agent 的方法，可能是不做 Agent。

先做 workflow，然后等真的客户进来了，再去把这个工作流打磨好。
赚了足够多的钱以后，再给用户做一个 Agent，然后作为一个 marketing 的东西。

B.ai 是表单，然后另外两个竞品不是这样的，都是自己做了个 chat 界面，但是我去看了他们所有的 Discord 的东西。B.ai 的抱怨非常少，而另外两个竞品用了 chat 界面的都很多，很多人说这个也不对劲，那个也不对劲。

我自己做了三个星期 Agent，我发现我也调不好。用户只是要搞一个 MV，那个聊天框的唯一作用就是一个工作流的调度，它没有什么其它的功能。反而引入了很多问题。

最后成绩相对来讲可能还是 B.ai 好一些，说明用户喜欢。另外两个竞品通过一个非常巨大的开发成本，最后得到了和他差不多的成绩，还有一堆用户抱怨。

所以说，正确地做 Agent 的方法，可能就是不做 Agent。

两个竞队的抱怨基本上都是：
1. 这个突然又不好用了
2. 那个突然又不好用了
3. 或者说 Agent 在打圈圈了，没有办法去干预

那他们的界面就是没有办法让人去干预的。因为它是 Chat First 的界面，Chat 是第一位的，然后用户没有办法搞。然后我们从表单做起，就是 Human-first，是我们自己可以操作的，永远是可控的，因为 Agent 会出问题。宁可我这个东西验证了，赚到钱了之后，最后再往上加 Agent。给用户一个很哇塞的感觉，然后出去宣传。

进一步来说，我甚至一开始就可以宣称自己是 Agent，写都不用写了。

---

我感觉 Agent 一定是 Human First 的。也就是说，它一定得是一个人一开始就能用的产品，让人先喜欢上它，最后再往上加一点点自动化。要不就是像 Suno 这种，它做了 Suno Chat，这是一个非常简单的流程，所以可以这么搞。

复杂的流程不行。你看那些什么 Pollo 啊什么的，他们都在招 Agent 的工程师，为什么？因为他有一个很大的盘子，他也知道用户怎么用，我们小卡拉米起来是不行的。

那也就是说，任何的人用的产品，其实都可以 Agent 化，在外面外挂就是了。我们不是需要去研究一些 Agent 原生的产品。那我懂的，人用的产品是什么呢？不就是 Video All-in-One 站吗。

在这个站上，甚至我都不是很想去搞 MV 生成了。就做一个 All-in-One 的站，然后配上一个 Skill，把它调好，确保每一个 OpenClaw、claude code 这些都能接进去。然后用户本地爱搞什么搞什么，爱用多少 token 就用多少 token。我现在看起来，好像没有什么人在这么搞，大家都是想在自己的产品里塞进一个 Chat。

而且这个想法更妙的事情是，我只需要零个奇迹，我连一个奇迹都不需要。即使这个想法是不对的，那它还是一个 All-in-One 站呀，我该怎么运营就怎么运营。然后，我的宣传语就可以是 Agent Native Video All In One。或者说 Agent First。

总之就是一个想法：最好的 Build Agent 的方法是不用 Build Agent，先做一个普通人能用的产品。那我知道的普通人能用的产品就是这个 Video All In One，我也知道怎么去做。然后我们要把接口写好，在建站的第一天，就要考虑到怎么样用 Skill 去调，然后把这个东西也放出去。

就我们不做 Agent，只做他需要的一些多媒体的搬运工。至于什么 MV 生成计划这类东西，那干脆都让 Cloud Code 或者 OpenClaw 去做，他们做的肯定比我自己手搓的工作流棒多了。我可以自己给一个官方的，我可以调好，然后让用户去做呗。

如果说他自己没有 Claude Code 或者 CC，那我网站上也可以挂一个 chat 界面，就是简单一点，收点钱呗。

这样子的话，很多脏活累活都从我这里丢出去了。我也不用开发 Agent，我也不用调优，真的也调不好。而且最终用户用的效果都差不多。

---

Agent 是可以在你的产品之外的，也应该就是在你的产品之外。

因为重新开发一个 Agent，你打得过 Claude 吗？你打得过那些吗？再提一个暴论：

在产品里做 Chat 的厂商都想做一个“围墙花园”，让你留在他的站内使用，这是“大哥”们的玩法。

但你们做这些，最后其实都做不过 Cloud Code（CC）和那些大厂的。只要你敢开接口，（逻辑就会完全不同）。

拿 Suno 做例子，Suno 把 chat 做在里面，那它肯定不会开接口呀。我网上搜了，也没有说 Suno Skill 这种东西，有一两个好像也被它版权炮一下子打掉了。

但是我自己做一个 All-in-One，我没有这个问题。我就是逆向的 Suno，然后包装成我自己的产品名字卖出去。我是可以去掉（Suno 痕迹）的，我自己的网站我不在乎，因为我只是个搬运工。只要我不说这个是 Suno，我就说是我自己品牌的音乐模型就好了。这又是一个可以投机取巧的地方。

---

而且就像二哥说过，这不就是两个生态的边界吗？

Video 厂商和 Agent 厂商，在两个生态的边界，我做一个粘合剂的作用。

就像 Claude，它肯定不会去做视频和图片；而字节跳动也不一定会做得比 Claude Code 更好。就怎么说呢，我是能活着的。那些自己在产品里造一个 Agent，然后又敢开 API 的厂商，他一定会干不过 Cloud Code 这种通用的 Agent，加上调用的 Skill。

你自己一人之力，怎么可能敌得过网上所有人呢？

那我干脆就做 Agent-first，我就做多媒体的搬运工，我打不过就加入。我不是来拆散这个家的，我是来加入这个家的。

然后另一个问题就是：比如说 WaveSpeed 之类的 API 厂商做了怎么办呢？好办啊，我卖给他或者让他投资我呗。而且市场也够分，我这个小卡拉米活下来也不成问题，更可能是完全注意不到。

至少现在好像没有说一个厂有一个 skill，skill 很多人都是自己写然后接到厂商 api，我现在知道很多做音视频的这些创作者去调那些厂商的接口，skill 我就帮他们把这“最后一公里”做了，肯定会有一部分人买单的，无论是他做得累，还是说他没这个时间。
