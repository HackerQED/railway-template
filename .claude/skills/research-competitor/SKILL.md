# 竞品调研

## 开始之前（必须）

**先问，再做。**

1. 问用户：调研这个竞品的目的是什么？是看产品设计、定价、技术架构、还是营销话术？
2. 问用户：有没有特别想关注的维度？有没有你已经知道的背景信息？
3. 自己主动想：这个调研还有什么遗漏的地方？用户可能没想到但值得看的角度？提出来让用户确认。

不要拿到 URL 就开始爬。方向错了，做再多也白费。

## 第一步：存档

**先存档，再分析。** 竞品随时会改版、下架、调整定价。当前状态是最珍贵的原始材料。

存档内容（按优先级）：
1. **截图** — 关键页面全页截图，Markdown 引用。竞品改版后截图是唯一证据
2. **可下载的产出物** — Skill 源码、API 文档、SDK 包等。保留原文，不读了再写，直接拿
3. **结构化快照** — 用 `browser_snapshot` 拿 accessibility tree，信息量比截图大

存档目录结构：
```
{调研目录}/
├── README.md              # 总览 + 横向对比
└── {竞品名}/
    ├── README.md          # 单个竞品的调研报告
    ├── *.png              # 截图（Markdown 引用）
    └── {产出物}/          # 原文保留（Skill 源码、API 文档等）
```

存档位置由用户指定。如果用户没指定，问一下放哪里。

## 第二步：拆解（批判式）

**营销话术和实际能力是两回事。**

对每条信息，问三个问题：
1. **这是事实还是吹牛？** — "专业级"、"电影级"、"唯一"这些词都是营销，不是事实
2. **能从源码/行为验证吗？** — 看源码推断实际能力，比信首页文案靠谱得多
3. **这个措辞我们能用吗？** — 好的营销话术可以借鉴，标注为"可借鉴的术语"

### 拆解维度

根据调研目的选择相关维度，不需要每次全做：

| 维度 | 要看什么 | 怎么验证 |
|------|---------|---------|
| 产品定位 | 面向谁、核心叙事 | 看首页 H1 和 CTA |
| 实际能力 | 能做什么、做多快 | 看源码、API 设计、架构 |
| 定价 | 怎么收费 | 看定价页，算单位成本 |
| 技术架构 | 怎么实现的 | 看源码、Skill 设计、状态机 |
| 开放程度 | 围墙花园还是开放 | 看知识放哪里、用户能不能改 |

### 报告格式

每个竞品的 README.md：
- 营销话术摘录（标注"可借鉴的术语"和"不用当真的吹牛"）
- 实际能力（从源码/行为推断，和宣传对比）
- 对我们的参考价值（可借鉴 / 不用怕）

## 第三步：对比和策略

存档和拆解做完后：

1. **横向对比表** — 事实为主，营销话术已过滤
2. **关键发现** — 3-5 条，每条要有"所以呢"（对我们意味着什么）
3. **竞争策略** — 他们的弱点、我们在哪个局部能赢、护城河是什么

调研材料（客观）和策略判断（主观）分开放。策略文档放 `notes/`。

## 注意事项

- **不要被竞品的话吓到。** 批判地看，他们也是人做的，也有吹牛的时候。
- **重点是摘出我们能用的东西。** 好的术语、好的设计模式、好的定价策略。不是写一篇夸他们的报告。
- **存档要完整。** 竞品会改版。今天看到的东西明天可能就没了。能存的都存。
- **技术分析单独写 notes。** 如果拿到了源码，做源码级分析，这个价值最高。
